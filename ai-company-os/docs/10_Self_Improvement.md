# 10. Self Improvement(自己改善ループ)

## 原則

現状で満足しない。ただし改善も Issue 駆動で行う(勝手に大改造しない)。

## 改善ループ

1. **観測**: 作業のたびに、手作業だった箇所・失敗した箇所・迷った箇所を記録する。
2. **資産化**: 2回以上繰り返した手作業はスクリプトかテンプレートに変換する候補にする。
3. **提案**: 改善案を Issue 草案(効果 / 難易度 / リスク付き)として提出する。
4. **承認**: 人間が優先順位を決める。承認境界に触れる改善は必ず承認を待つ。
5. **反映**: 実装後、該当する `docs/` を同じ PR で更新する(ドキュメントと実態を乖離させない)。

## 定期レビュー(月次目安)

- `docs/03_Development.md` の「既知事実」を再検証する(環境は変わる)。
- 採用中ツールの更新状況・セキュリティを確認する(判定の見直し: adopt → watch など)。
- テンプレートの使用実績を確認し、使われていない項目を削る。

## 改善候補の評価軸

| 軸 | 問い |
|---|---|
| 効果 | 1回あたり何分節約できるか。品質は上がるか |
| 難易度 | 実装は1 Issue に収まるか |
| リスク | 承認境界・秘密情報・外部依存に触れるか |
| 再現性 | 環境が変わっても動くか |
