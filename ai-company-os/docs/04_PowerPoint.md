# 04. PowerPoint Standards(PowerPoint 制作標準)

## 基本仕様

- スライドサイズ: 16:9(13.333 × 7.5 インチ)
- フォント: 游ゴシック(Yu Gothic)を基本。英数は同フォントで統一
- 1スライド1メッセージ。本文は最大5ブレット、1ブレット最大2行
- 全スライドにスピーカーノート必須(話す内容の完全な下書き、1枚あたり100〜300字)

## 配色

パレットは3色+背景で固定し、デッキ内で混在させない。`build_deck.py` には3つの既定プリセットがあり、
`meta.palette_preset` で選択する(個別の色は `meta.palette` で上書き可能):

| プリセット | Primary | Base | Accent | 向いている用途 |
|---|---|---|---|---|
| `navy_gold`(既定) | `#1E2761` | `#CADCFC` | `#D9A441` | 信頼・フォーマル(営業・提案資料) |
| `forest_sand` | `#24523F` | `#E7DCC2` | `#C1622D` | サステナ・自然派・教育 |
| `slate_azure` | `#2B3A55` | `#D3E4F5` | `#1F8FD6` | テック・プロダクト紹介 |

## 構成の型・標準枚数(2026-07-23〜、30枚を標準とする)

オーナー指示により、**1テーマあたり30枚を標準の目標枚数**とする(大幅に下回らないこと。
`verify_pptx.py --min-slides 28 --max-slides 32` を目安に検品する)。ただし後述の
「動画としてのテンポ」ルールに従って1項目=1スライドに分解した結果30枚を超える場合は、
下回るよりも超える方を優先してよい(目安30〜45枚)。単なる11枚の型を
3倍に薄めるのではなく、下記のように「導入・基礎知識・数字による深掘り・活用・注意点・
まとめ」まで章立てを増やして厚みを持たせる(実例: `assets/2026-07-21_claude-sonnet5-cost-simulation/deck_spec.json`)。

1. タイトル(フックあり) 2. 今回の結論 3. この動画で得られること 4. なぜ今扱うのか
5〜8. 背景データ・報道され方・時系列(agenda) 9〜11. 基礎知識・用語解説・比較(以前 vs 現在)
12〜19. 重要ポイントを数字で深掘り(cards/table/bar_chart等) 20〜27. 活用方法・成功/失敗パターン・
注意点・反対意見弱点 28〜30. 今後の予測・CTA・出典一覧

内容によっては「比較」を `comparison`(2軸比較)や `table`(価格表・マトリクス)に、
「アジェンダ」を `agenda` に、証言・引用の挿入に `quote` を使うなど、下記レイアウトを組み合わせる。

## エンゲージメント設計ルール(離脱防止、2026-07-23〜)

`research/2026-07-23_audience-retention-techniques.md` の調査に基づく。

- **冒頭フック**: 表紙(スライド1)は挨拶・自己紹介だけで終わらせず、結論の一部や
  驚きの数字を匂わせるサブコピーを入れる
- **レイアウトの多様性(パターンインタラプト)**: 同一レイアウトを4連続以上使わない
  (`build_deck.py`がビルド時に検証・拒否する。3連続までは許容)
- **数字は見せる**: 数字を話すスライドは必ずchart/table/cards/big_statのいずれかで
  視覚化する(bulletsで数字を羅列するだけにしない)
- **約束→提供→次の約束**: agendaスライドで先の展開を予告し、各章の最後に次章への
  一言(オープンループ)を入れる

## 動画としてのテンポ・視覚デザイン(2026-07-23〜、オーナー指示)

「もっと見たくなるように」「画面の情報量は少なく、どんどんスムーズに流れていく感じに
したい」という指示を受けたルール。デッキは常にナレーション動画
(`build_narrated_video.py`、`docs/13_AI_Dialogue_Script.md`参照)と対で作られる前提に
なったため、「1枚をじっくり読む資料」ではなく「1枚が1テンポの動画のコマ」として設計する。

- **1スライド1メッセージ**: `cards`が3〜4項目、`table`が5〜6行のように複数の要素を
  1枚に詰め込むのではなく、動画化する場合は**可能な限り1項目=1スライドに分解する**
  (例: 想定用途4件を1枚のcardsで見せるのではなく、big_stat/cards単体で4枚に分ける)。
  分解すると通常30枚の目安を超えるが、動画向けデッキでは超えてよい(目安30〜45枚)
- **ナレーション1行を短く**: `dialogue_spec.json`の各行は1〜2文以内の短い一言にする
  (長い説明を1行に詰め込まない)。スライド1枚あたりの表示時間が短くなり、動画として
  次々に場面が切り替わるテンポが生まれる
- **視覚要素は既定でアクセントバー付き**: `build_deck.py`は見出し下にアクセントの
  短い下線(`title_bar`)、カードには右下にずらした影(`card`)、スライド最上部には
  全体に対する進捗を示す細いバー(`progress_bar`、`meta`に応じて自動計算)を必ず描く。
  `render_deck_images.py`(動画用の画像描画)も同じ見た目になるよう常に同期させる
- 上記は`build_deck.py`/`render_deck_images.py`の両方に実装済みで、個別のtoggleは無く
  常に適用される(pptx単体で見ても、動画にしても同じ見た目になることを優先している)

## 文字を減らし、見て分かるスライドにする(2026-07-23〜、オーナー指示)

「文字が多いので、図形・グラフでもっと見てわかるようにしてほしい」という指示を受けた
ルール。**bullets(文字の箇条書き)はデッキ全体の40%までとし、超えていれば
`build_deck.py`がビルド時にエラー終了する**(10枚以上のデッキが対象)。

内容の性質ごとに、bulletsではなく以下の視覚的レイアウトへ積極的に置き換える:

| 話したい内容 | bulletsの代わりに使うレイアウト |
|---|---|
| 単一の衝撃的な数字を強調したい | `big_stat`(スライド全体で1つの数字だけを見せる) |
| 複数の数字・指標を並べたい | `cards`(数字+ラベル+短文を2〜4枚) |
| 概念同士の関係・つながりを説明したい | `diagram`(ノードを矢印でつなぐ簡易概念図) |
| 手順・プロセスを説明したい | `steps`(矢印つきの横並びステップ) |
| 2つの立場・選択肢を比べたい | `comparison`(2カラム比較) |
| 複数項目×複数属性を整理したい | `table` |
| 推移データ・構成比を見せたい | `line_chart` / `pie_chart` / `bar_chart` |

bulletsを使ってよいのは、上記のどれにも当てはまらない純粋な要点整理(不明点・注意点の
一覧など)に限る。1スライドの本文は最大5ブレット・1ブレット最大2行という既存ルールは
維持しつつ、そもそもbullets自体を安易な既定選択にしない。

## 初心者にもわかる・実践につながる構成にする(2026-07-23〜、オーナー指示)

「初心者でもわかるように、テーマによってどうなるのか・どう使えるのかもわかるように」
という指示を受けたルール。専門用語を並べるだけの解説で終わらせず、以下の2点を
どのテーマでも必ず満たす:

- **初心者向けの前提知識**: 専門用語(例:「AIエージェント」「エージェント基盤」等)が
  出てくるテーマでは、本題に入る前に1〜2枚使って、その用語が何を指すのか・なぜ今の
  文脈で重要なのかを平易な言葉で説明する
- **「これで何が変わるか・どう使えるか」セクション**: 起承転結の「転」または「結」に、
  そのニュースが読み手(個人・企業)にとって具体的に何を意味するか、実際に検討・
  活用するとしたら何を確認すべきかを扱うスライドを入れる。単なる出来事の紹介で
  終わらせない

なお、**前サイクルのテーマとの関連づけ(「前回テーマとの共通点」のようなスライド)は
2026-07-23より行わない**(オーナー指示)。各テーマは単体で完結させる。

## データと出典

- 数字を出すスライドには必ず出典をフッターまたはノートに記載する。
- グラフは python-pptx のネイティブチャート(bar_chart / line_chart / pie_chart)で生成する。画像貼り付けのグラフは編集不能になるため避ける。
- 推移データは `line_chart`、構成比は `pie_chart`(カテゴリ5個以下推奨)、量の比較は `bar_chart` を使い分ける。

## 生成方式(2026-07-23〜、オーナー指示により再度デフォルト運用)

2026-07-21〜23の期間は生成方式B(ChatGPTプロンプト方式)がデフォルトだったが、
オーナーの「もう送らないのであなたがすべて開発してください」という指示により、
**2026-07-23以降は生成方式A(`build_deck.py`による直接生成)を再びデフォルトとする**。
以後のテーマ制作(定期実行含む)では、ChatGPT投入用プロンプトだけでなく、
実際に`.pptx`ファイルまで生成し、検品・原稿出力まで済ませてから納品する。
生成方式Bは、オーナーから明示的な指示があった場合のみ使う補助手段とする。

## 生成と検品(標準パイプライン)

```text
① 調査する        Web検索等で最新情報を集め、出典を記録する(docs/06_Content_R&D.md準拠)
② 構成を書く      templates/deck_outline_template.md(人間が読める構成+ノート)
③ JSON仕様にする  templates/deck_spec_example.json を複製して書き換える
④ 生成する        python scripts/build_deck.py spec.json out.pptx
⑤ 検品する        python scripts/verify_pptx.py out.pptx
⑥ 原稿を出す      python scripts/export_script.py spec.json script.md
```

`export_script.py` は同じ JSON 仕様からスピーカーノートを「読み上げ原稿」として
Markdown に書き出す(スライドごとの見出し・全文ノート・目安発表時間・合計時間)。
PowerPoint を開かずに原稿だけを読んで推敲・リハーサルできる。原稿にノート欠落があれば
`build_deck.py` と同じ規則でエラー終了する。

- `build_deck.py` は 14 レイアウト(title / bullets / cards / steps / agenda / comparison / table /
  quote / bar_chart / line_chart / pie_chart / cta / big_stat / diagram)に対応し、ノート欠落・
  ブレット6個以上・table の列数不一致・bullets比率40%超過をビルド時にエラーとして拒否する。
  全レイアウトの実例は `templates/deck_spec_example.json`(14スライド)を参照。
- 特殊なレイアウトが必要な場合のみ個別スクリプトを書き、スクリプトをリポジトリに残す。
- 生成後は必ず `python scripts/verify_pptx.py <file>` を実行する。検品項目:
  - スライド枚数が構成と一致
  - 全スライドにノートが存在
  - 図形がスライド境界からはみ出していない
  - 空のプレースホルダが残っていない
- レンダリング(見た目)検証ができない環境では、その旨を成果報告に明記する。

## 生成方式B: ChatGPTプロンプト方式(2026-07-21〜、オーナー指示によりデフォルト運用)

`build_deck.py` による直接生成の代わりに、**調査結果を反映したChatGPT投入用プロンプトを成果物とする**方式。
オーナーが個人のChatGPTアカウントに貼り付けて生成する運用のため、pptxファイル自体は作らない。

```text
① 調査する    Web検索で情報収集し、research.md に検証済み事実/推測/不明点を分けて記録する
② プロンプト化 chatgpt_prompt.md に、出典付きの事実・スライド構成・デザインルールを
               1つのプロンプトとしてまとめる(コピペ即使用できる形)
③ 納品する    research.md と chatgpt_prompt.md をユーザーに送付する
```

プロンプトに必ず含める要素:

- 「必ず使う事実」として出典付きの数字・事実を列挙し、範囲外の数字を創作させない一文を入れる
- 「不明な点」として調査で確認できなかった項目を明記し、推測で埋めさせない
- スライド構成(枚数目安・各スライドの役割)
- 本ドキュメントのデザイン・品質ルール(16:9、ノート必須、ネイティブグラフ、出典明記等)
- 出力形式の指定(Code Interpreterが使えるなら実際に .pptx を生成させる、使えないならテキスト出力)

`build_deck.py` 方式(生成方式A)は、CI検証込みで再現性の高い定型フォーマットが必要な場合に引き続き使う。
どちらの方式を使うかは、その時々の指示に従う。

## 公式スキルとの役割分担

Claude Code には Anthropic公式 `anthropics/skills` 由来の `pptx` スキルが標準搭載されている(source-available)。
`build_deck.py` を自前で持つ理由:

- 公式スキルの利用条件変更の影響を受けない、再現可能な生成ロジックを持つため
- ノート必須・ブレット上限などの**このリポジトリ固有の品質規則をコードで強制**するため(公式スキルは汎用)
- CI(`ai-company-os-ci.yml`)で毎回 E2E 検証できる形にするため

単発の凝ったデザインが必要な場合は公式 `pptx` スキルを使い、標準フォーマットの反復生産には `build_deck.py` を使う。
詳細な比較は `research/2026-07-20_useful-repositories.md` を参照。
